# 存储模块说明

本文档用于说明 KiCloud 当前版本的文件存储模块实现，包括路由、服务职责、目录树模型、单文件 / 分片上传策略、容量控制、删除策略与下载校验规则。

---

## 1. 模块概览

当前存储模块已经支持以下能力：

1. 上传单个完整文件
2. 上传大文件分片并在服务端合并
3. 用户控制台一次选择多个文件并顺序上传
4. 创建文件夹
5. 查看当前目录文件列表
6. 按关键字搜索文件 / 文件夹
7. 重命名文件或文件夹
8. 删除文件或空文件夹
9. 下载自己的文件
10. 查看当前账号存储空间摘要
11. 超级管理员查看指定用户的文件列表与搜索结果

主要入口为：

- `src/main/java/club/muimi/kicloud/controller/StorageController.java`
- `src/main/java/club/muimi/kicloud/service/StorageFileService.java`
- `src/main/resources/templates/dashboard.html`

## 2. 相关类与职责

### 2.1 StorageController

代码位置：`src/main/java/club/muimi/kicloud/controller/StorageController.java`

职责：

- 提供 `/storage/**` 路由入口
- 从 Session 中读取 `LoginUser`
- 对请求对象是否为空做基础校验
- 将具体业务交给 `StorageFileService`

当前接口包括：

- `POST /storage/upload`
- `POST /storage/uploadSlices`
- `POST /storage/uploadSlice`
- `POST /storage/folder`
- `GET /storage/list`
- `GET /storage/search`
- `POST /storage/rename`
- `POST /storage/delete`
- `GET /storage/summary`
- `GET /storage/download`

### 2.2 StorageFileService

代码位置：`src/main/java/club/muimi/kicloud/service/StorageFileService.java`

职责：

- 校验登录用户合法性
- 校验父目录是否合法
- 处理单文件上传并写入数据库记录
- 创建分片上传会话并接收单个切片
- 合并全部切片并执行完整文件 MD5 校验
- 定时清理过期分片会话和临时切片文件
- 创建文件夹记录
- 查询目录列表与搜索结果
- 校验重命名与删除规则
- 处理用户容量统计
- 处理文件下载响应
- 为分享模块提供“可公开下载文件”的统一校验入口

### 2.3 StorageFileDao

代码位置：`src/main/java/club/muimi/kicloud/dao/StorageFileDao.java`

职责：

- 查询未删除文件 / 文件夹
- 按用户、父目录、名称执行去重与列表查询
- 支撑搜索、兄弟节点重名判断、父目录校验等逻辑

### 2.4 StorageFile 实体

代码位置：`src/main/java/club/muimi/kicloud/entity/StorageFile.java`

职责：

- 表示文件或文件夹节点
- 通过 `fileType` 区分 `FILE` 与 `FOLDER`
- 通过 `parentId` 维护目录树
- 通过 `storageKey` 关联物理文件内容

### 2.5 分片上传模型

代码位置：

- `src/main/java/club/muimi/kicloud/model/Slice.java`
- `src/main/java/club/muimi/kicloud/model/Slices.java`
- `src/main/java/club/muimi/kicloud/model/UploadSlicesRequest.java`
- `src/main/java/club/muimi/kicloud/model/UploadSliceRequest.java`

职责：

- 用 `Slices` 表示一个完整文件的上传会话
- 用 `Slice` 表示单个切片的完成状态
- 用 `UploadSlicesRequest` 承接“开始分片上传”的请求
- 用 `UploadSliceRequest` 承接“上传单个切片”的请求

## 3. 数据与目录模型

### 3.1 统一节点模型

当前系统不拆分“文件实体”和“文件夹实体”，统一使用 `StorageFile` 表示。

字段上的关键约定：

- `fileType = FILE`：表示普通文件
- `fileType = FOLDER`：表示文件夹
- `parentId = null`：表示根目录
- `ownerId`：表示所属用户
- `deleted = true`：表示逻辑删除

### 3.2 目录树由数据库维护

当前目录结构不依赖磁盘真实目录树，而由数据库字段 `parentId` 维护。

这意味着：

1. 文件夹节点主要用于组织目录
2. 真实磁盘上不一定为每个文件夹创建对应目录
3. 文件与文件夹的显示层级完全由数据库决定

### 3.3 分片会话由内存维护

分片上传过程中的会话状态不写入数据库，而保存在 `StorageFileService` 的 `activeSlices` 内存映射中。

这意味着：

1. 分片上传任务只在当前应用进程内可见
2. 服务重启后未完成的分片上传任务不会恢复
3. 会话超时清理依赖服务内的定时任务和接口入口触发

## 4. 存储根目录与临时目录

### 4.1 正式文件存储根目录

配置项：`kicloud.storage.root=./data/storage`

当前服务会在启动时把该路径解析为绝对标准路径，并在上传 / 下载时统一基于该根路径进行路径拼接与校验。

#### storageKey 生成规则

当前生成格式为：

```text
{userId}/yyyy/MM/{timestamp}_{userId}_{suffix}{ext}
```

这意味着：

1. 同一用户的文件会按用户 ID 分区
2. 再按年月分层
3. 最终文件名由时间戳、用户 ID 与随机后缀构成
4. 原始文件名不会直接作为真实磁盘文件名

### 4.2 分片临时目录

分片上传使用的临时目录默认为：

```text
./data/temp
```

该路径来自 `StorageFileService` 中的 `@Value("${kicloud.storage.temp-root:./data/temp}")` 默认值。

当前约定：

- 单个切片文件名：`{ownerId}-{uploadId}-{sliceNum}.part`
- 合并中的临时完整文件名：`{ownerId}-{uploadId}.merged`

### 4.3 路径安全校验

上传、分片落盘、合并和下载时都会校验最终路径是否仍位于对应根目录之内，避免路径拼接导致越界访问。

## 5. 前端上传行为

代码位置：`src/main/resources/templates/dashboard.html`

当前用户控制台中的上传表单保留原有交互方式：

- 仍然使用同一个文件选择框
- 仍然使用同一个“上传到当前目录”按钮
- 仍然通过原消息框展示进度与结果

但其行为已经扩展为：

1. 文件选择框支持一次选择多个文件
2. 多个文件会按选择顺序逐个上传
3. 当文件大小 `<= 10MB` 时，走原 `POST /storage/upload`
4. 当文件大小 `> 10MB` 时，先发起 `POST /storage/uploadSlices`，再循环调用 `POST /storage/uploadSlice`
5. 分片上传时，前端先计算完整文件 MD5
6. 单个切片上传失败时，前端会自动重试；连续 5 次都失败才终止该文件上传

## 6. 单文件上传流程

接口：`POST /storage/upload`

请求参数：

- `file`
- `parentId`（可选）

`StorageFileService.uploadFile(...)` 当前主要流程如下：

1. 校验 `LoginUser` 对应数据库用户是否仍然存在、启用且身份一致
2. 校验上传文件对象存在且不为空
3. 清理原始文件名
4. 校验文件名合法
5. 若传入 `parentId`，校验父目录存在且必须是当前用户的文件夹
6. 检查同一父目录下是否已存在同名未删除节点
7. 校验文件大小大于 `0`
8. 根据角色确定单文件上传上限
9. 校验 `usedSpace + fileSize <= totalSpace`
10. 生成 `storageKey`
11. 创建目标磁盘目录
12. 将文件写入磁盘
13. 创建 `StorageFile` 记录
14. 增加用户 `usedSpace`
15. 返回上传成功结果

---

## 7. 分片上传流程

### 7.1 开始分片上传

接口：`POST /storage/uploadSlices`

请求体：`UploadSlicesRequest`

关键字段：

- `sliceCount`
- `fullFileMD5`
- `fileName`
- `parentId`
- `totalSize`

当前流程：

1. 校验登录用户合法性
2. 先主动清理一次已过期分片会话
3. 校验切片数、完整文件 MD5 和完整文件大小
4. 复用正式上传的文件名、父目录、重名、单文件上限和总容量校验
5. 创建 `./data/temp` 目录
6. 生成独立 `uploadId`
7. 创建 `Slices` 会话对象并存入 `activeSlices`
8. 若同一个 `uploadId` 已有旧会话，则先清理旧切片文件
9. 返回 `uploadId` 给前端继续上传单个切片

### 7.2 上传单个切片

接口：`POST /storage/uploadSlice`

请求体：`UploadSliceRequest`

关键字段：

- `uploadId`
- `fullFileMD5`
- `sliceNum`
- `size`
- `slice`

当前流程：

1. 校验登录用户合法性
2. 先主动清理一次已过期分片会话
3. 校验切片内容、切片编号和切片大小
4. 校验完整文件 MD5 与 `uploadId`
5. 按当前用户和 `uploadId` 定位会话
6. 校验该会话没有过期，且 `fullFileMD5` 与会话一致
7. 将切片写入 `./data/temp/{ownerId}-{uploadId}-{sliceNum}.part`
8. 更新会话中的对应 `Slice`
9. 若未全部完成，则返回继续等待上传
10. 若已全部完成，则转入切片合并流程

### 7.3 全部切片完成后的处理

当 `Slices.isAllDone()` 返回 `true` 后，服务端会：

1. 按切片编号顺序读取全部切片文件
2. 合并成临时完整文件
3. 重新计算合并后文件的 MD5
4. 与前端最初提交的 `fullFileMD5` 进行比对
5. 若一致，则复用正式单文件上传的保存规则生成 `storageKey`
6. 把合并后的完整文件移入正式存储目录
7. 创建 `StorageFile` 记录并增加用户已使用容量
8. 无论成功还是失败，都删除全部切片文件和临时合并文件

### 7.4 会话唯一标识

当前分片上传不再仅依赖 `ownerId + fullFileMD5` 识别会话，而是使用：

```text
ownerId + uploadId
```

这样可以支持：

- 同一用户同时上传两个内容完全相同的文件
- 相同 MD5 文件在不同目录下并发上传

## 8. 分片会话超时与清理

当前系统规定：如果一个 `Slices` 会话在以下任一条件满足时仍未完成上传，则视为过期：

1. `createdAt + 20 分钟`
2. `updatedAt + 2 分钟`

当前清理机制有两层：

### 8.1 定时清理

`StorageFileService.cleanupExpiredSliceUploads()` 使用：

```text
@Scheduled(fixedDelay = 60000L)
```

即每 60 秒检查一次过期会话，并删除：

- `activeSlices` 中的会话对象
- `./data/temp` 中对应的切片文件

### 8.2 上传入口主动清理

每次调用：

- `uploadSlices(...)`
- `uploadSlice(...)`

之前，也会主动执行一次过期会话清理，避免过期任务持续占用临时文件。

## 9. 文件名清理与合法性约定

当前服务会对用户提交的文件名 / 文件夹名进行清理与限制，核心目标是避免：

- 空文件名
- 非法路径片段
- 目录穿越
- 由路径分隔符造成的伪造目录结构

当前约定包括：

1. 使用 `Paths.get(rawName).getFileName().toString().trim()` 取最终文件名
2. 拒绝空白名称
3. 拒绝 `.` 与 `..`
4. 拒绝包含正斜杠或反斜杠的名称

## 10. 上传大小限制与容量控制

### 10.1 Spring 请求级上传限制

配置文件中：

- `spring.servlet.multipart.max-file-size=10GB`
- `spring.servlet.multipart.max-request-size=10GB`

这是 HTTP 请求层面的硬上限。

### 10.2 角色级单文件上传限制

`StorageFileService` 中额外定义了角色级限制：

| 角色 | 单文件上限 |
|---|---|
| `USER` | 200MB |
| `ADMIN` | 1GB |
| `SUPERADMIN` | 10GB |

分片上传最终仍按完整文件大小校验上述限制，而不是按单个切片大小绕过限制。

### 10.3 用户总容量控制

上传前还会校验：

```text
usedSpace + fileSize <= totalSpace
```

若超出总容量，则上传失败。

## 11. 创建文件夹流程

接口：`POST /storage/folder`

请求体：`CreateFolderRequest`

当前流程：

1. 校验登录用户合法性
2. 校验文件夹名称不为空且合法
3. 若指定父目录，则校验父目录存在且为当前用户的文件夹
4. 检查同一父目录下是否存在同名未删除节点
5. 创建 `fileType = FOLDER` 的 `StorageFile` 记录
6. 当前文件夹大小保存为 `0L`

文件夹当前只是目录节点，不在磁盘中写入实际内容。

---

## 12. 列表与搜索

### 12.1 当前目录列表

接口：`GET /storage/list`

参数：`parentId`（可选）

规则：

- 若 `parentId` 为空，则查询根目录
- 若 `parentId` 不为空，则必须是当前用户的有效文件夹
- 只返回当前用户、未删除的节点

### 12.2 关键字搜索

接口：`GET /storage/search`

参数：`keyword`

规则：

- 关键字不能为空
- 当前只搜索当前登录用户自己的未删除节点
- 返回文件列表与当前用户存储空间信息

---

## 13. 重命名规则

接口：`POST /storage/rename`

请求体：`RenameStorageFileRequest`

当前流程：

1. 校验登录用户合法性
2. 校验目标文件 ID 不为空
3. 校验新名称合法
4. 查找当前用户名下的目标未删除节点
5. 检查同一父目录下是否已有其他同名未删除节点
6. 更新名称并保存

当前重命名不区分文件与文件夹，统一按节点名称修改。

---

## 14. 删除规则

接口：`POST /storage/delete`

请求体：`DeleteStorageFileRequest`

### 14.1 删除文件夹

当前规则：

- 只能删除空文件夹
- 如果目录下仍有未删除子节点，则返回失败

### 14.2 删除普通文件

当前规则：

1. 删除前先根据 `storageKey` 定位真实文件
2. 尝试删除磁盘中的物理文件
3. 扣减用户 `usedSpace`
4. 将数据库记录标记为：
   - `deleted = true`
   - `deletedAt = now()`

### 14.3 当前删除策略

当前对 `StorageFile` 记录采用**软删除**，但对物理文件内容采用**真实删除**。

这意味着：

- 元数据还能保留删除痕迹
- 文件内容本身不会进入回收站
- 当前版本尚不支持恢复已删除文件内容

---

## 15. 下载规则

接口：`GET /storage/download?id=...`

当前流程：

1. 校验登录用户合法性
2. 校验文件 ID 存在
3. 查询当前用户名下未删除节点
4. 校验该节点必须是普通文件
5. 校验 `storageKey` 存在
6. 解析实际文件路径
7. 再次校验目标路径仍在存储根目录内
8. 校验物理文件存在且为常规文件
9. 返回下载响应

当前下载只允许用户下载自己的文件。

---

## 16. 超级管理员扩展能力

除了普通用户自己的文件管理外，当前还提供：

- `GET /admin/user/{userId}/files`
- `GET /admin/user/{userId}/files/search`

这两个接口由 `AdminService` 转发给 `StorageFileService` 处理，但要求操作者必须是 `SUPERADMIN`。

因此：

- `ADMIN` 无法浏览其他用户文件
- `SUPERADMIN` 可以查看指定用户的目录与搜索结果

---

## 17. 与分享模块的协作

`StorageFileService` 还提供两个分享相关辅助能力：

1. `getActiveFileForLink(...)`
   - 校验目标节点存在、未删除、是普通文件、`storageKey` 有效、物理文件存在
2. `downloadSharedFile(...)`
   - 在分享下载场景下复用统一下载响应构建逻辑

这保证了分享模块和普通下载模块使用一致的底层文件校验规则。
